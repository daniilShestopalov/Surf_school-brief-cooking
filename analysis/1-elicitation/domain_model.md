# Описание доменной области (Domain Modeling)

Ниже представлено верхнеуровневое описание предметной области системы "Кулинарная студия" в контексте клиентского мобильного приложения, которое интегрируется с существующим бэкендом как "источником истины".

## 1. Основные сущности (Entities) и их ключевые атрибуты

* **Client (Клиент)** — физическое лицо, пользователь мобильного приложения, который ищет классы и бронирует места.
  * `id` — уникальный идентификатор.
  * `phone` / `email` — контактные данные (используются для авторизации).
  * `name` — имя пользователя.
  * `allergy_profile` — информация об аллергиях (если хранится глобально на уровне клиента).
  * `is_regular` — флаг постоянного клиента (для механик лояльности).

* **ClassProgram (Программа класса)** — шаблон кулинарного мероприятия с определенной темой, рецептами и уровнем сложности, который может проводиться многократно.
  * `id` — уникальный идентификатор.
  * `title` — название (например, "Итальянская кухня для новичков").
  * `description` — подробное описание и состав меню.
  * `complexity` — уровень сложности (новички / опытные).

* **Chef (Шеф)** — сотрудник студии или приглашенный повар, который ведет класс и обучает клиентов.
  * `id` — уникальный идентификатор.
  * `name` — имя или псевдоним.
  * `bio` — краткая информация (опционально).
  * `rating` — общая оценка шефа (рассчитывается бэкендом на основе отзывов).

* **Slot (Слот расписания)** — конкретное запланированное событие в расписании (проведение определенной программы), привязанное к дате, времени и шефу.
  * `id` — уникальный идентификатор.
  * `program_id` — ссылка на программу класса.
  * `chef_id` — ссылка на назначенного шефа.
  * `datetime_start` — дата и время начала класса.
  * `duration` — продолжительность (по умолчанию ~3 часа).
  * `max_capacity` — максимальное количество рабочих мест (от 8 до 12 в зависимости от программы).
  * `available_seats` — количество оставшихся свободных мест.
  * `status` — статус слота (доступен, отменен студией).
  * `address` — адрес проведения класса.
  * `available_equipment_stock` — доступный остаток прокатного инвентаря на данный слот.
  * `equipment_tariff` — стоимость аренды экипировки.

* **Booking (Бронирование)** — подтвержденный факт записи конкретного клиента на конкретный слот расписания, фиксирующий резерв рабочего места и опции (например, аренду экипировки).
  * `id` — уникальный идентификатор брони.
  * `client_id` — ссылка на клиента.
  * `slot_id` — ссылка на слот расписания.
  * `status` — статус записи (активна, отменена клиентом, отменена студией, завершена).
  * `needs_rental_equipment` — признак необходимости аренды экипировки (фартук, ножи).
  * `allergy_notes` — примечания по аллергиям к конкретному бронированию.
  * `chef_rating` — оценка, выставленная шефу после класса (может быть null).

## 2. Связи между сущностями (Relationships)

* **ClassProgram (1) — (N) Slot:** Одна образовательная программа может проводиться множество раз в разные даты и время.
* **Chef (1) — (N) Slot:** Один шеф может вести множество слотов (но на один слот назначен ровно один шеф).
* **Slot (1) — (N) Booking:** На один слот могут быть записаны несколько клиентов (в пределах `max_capacity`).
* **Client (1) — (N) Booking:** Один клиент может создавать множество бронирований на разные слоты.

## 3. Основные бизнес-процессы (High-level Use Cases)

1. **Просмотр расписания (Browse Schedule):**
   * Клиент открывает приложение и видит список доступных слотов по умолчанию на ближайшие 7 дней.
   * Если классов нет, отображается понятный пустой экран (Empty state).
   * Доступен фильтр дат для поиска классов за пределами базовых 7 дней.
2. **Бронирование места (Book a Slot):**
   * Клиент выбирает слот, знакомится с программой и шефом.
   * Клиент выбирает опции: требуется ли аренда инвентаря (если позволяет `available_equipment_stock`), указывает свои аллергии.
   * Приложение отправляет запрос в API; бэкенд гарантирует атомарную проверку (отсутствие двойных броней). При успехе бронь сохраняется в профиле клиента.
3. **Отмена бронирования клиентом (Cancel Booking):**
   * Клиент решает отменить свою запись. Приложение применяет бизнес-правила поздней/ранней отмены на основе ответов бэкенда.
4. **Обработка форс-мажора (Handle Studio Cancellations):**
   * При отмене класса по инициативе студии (срыв поставки и т.д.), статус брони клиента меняется на "Отменён студией".
   * Клиент получает Push-уведомление с указанием причины отмены; возможность повторной записи на этот слот блокируется.
5. **Оценка шефа (Rate Chef):**
   * После завершения слота клиент получает предложение (например, через Push-напоминание) оценить работу шефа. Клиент выставляет оценку, которая передается в API.

## 4. Глоссарий терминов

* **Класс (или Программа)** — методическое описание кулинарного мероприятия (тема, рецепты, сложность). Сама по себе программа не привязана ко времени.
* **Слот** — конкретное событие в расписании (проведение Класса в заданную дату и время, с определенным Шефом и пулом рабочих мест).
* **Бронь** — подтвержденный факт записи Клиента на конкретный Слот. Обеспечивает резерв рабочего места и прокатной экипировки.
* **Шеф** — ведущий Класса (штатный сотрудник или приглашенный специалист).
* **Прокатный фонд** — физический инвентарь (фартуки, наборы ножей), доступный для использования Клиентами во время Слота на условиях аренды.
