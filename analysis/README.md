# Проектное пространство: Бронирование кулинарных мастер-классов (Analysis)

Добро пожаловать в единое аналитическое пространство проекта. Здесь собрана вся проектная документация, от первоначального брифа заказчика до детальных контрактов API и спецификаций мобильного приложения.

Процесс проектирования построен по принципу **сквозной трассировки**. Это означает, что каждое бизнес-требование логически прослеживается через пользовательские сценарии к конкретным экранам, архитектурным решениям и, в конечном итоге, к методам API.

## 🔗 Схема трассировки (Как читать требования)

Документация построена таким образом, чтобы любой элемент можно было отследить от идеи до реализации. Взаимосвязь идентификаторов выглядит следующим образом:

1. **Бизнес-цель / Требование (BR / FR):** Фундаментальное требование (например, "Клиент должен иметь возможность забронировать место на мастер-классе").
2. **Сценарий (UC):** Вариант использования, описывающий шаги пользователя для достижения цели. Ссылается на BR/FR.
3. **Экраны и Шторки (SCR / BS):** Дизайн-брифы (в папке `3-design-brief`), описывающие визуальное представление шагов из UC.
4. **Спецификация UI/Логики (SCREEN / LOGIC):** Детальные технические спецификации для разработчиков (в папке `5-mobile-app-spec`), жестко привязанные к ID экрана (например, `SCR-004_slot_details_SCREEN.md`).
5. **Контракт API (operationId):** Endpoint в `api/openapi.yaml`, который реализует логику получения или отправки данных, описанную в файлах `_LOGIC`.

**Пример сквозного пути:**
`FR-03 (Бронирование)` ➔ `UC-02 (Запись на класс)` ➔ `SCR-004 (Детали слота)` ➔ `SCR-004_slot_details_LOGIC.md` ➔ `operationId: bookSlot` (POST /bookings).

## 👥 Матрица документов (Кому что читать)

Шпаргалка для команды: какие папки содержат наиболее важную для вашей роли информацию.

* **💻 Разработчики мобильного приложения (iOS/Android):**
  В первую очередь ваш рабочий контекст находится в папке **[5-mobile-app-spec](./5-mobile-app-spec/)**. Для понимания общей картины данных заглядывайте в **[api](./api/)** и изучите **[Шаг 4 (Архитектура)](./4-design/)**.
* **⚙️ Backend-разработчики:**
  Ваш основной фокус — контракты **[api](./api/)** и архитектурные схемы **[Шаг 4](./4-design/)**. Для понимания того, как данные используются на клиенте, сверяйтесь с бизнес-логикой в **[Шаг 5](./5-mobile-app-spec/)**.
* **🎨 UI/UX Дизайнеры:**
  Ваша зона ответственности — **[Шаг 3 (Дизайн-брифы)](./3-design-brief/)**. Обязательно сверяйтесь с **[Шагом 2 (Требования)](./2-requirements/)** для понимания всех необходимых пользовательских сценариев.
* **🐞 QA Инженеры (Тестировщики):**
  Для написания тест-кейсов вам потребуется вся цепочка: от **[Шага 2 (Use Cases)](./2-requirements/)** для E2E тестов, до **[Шага 5](./5-mobile-app-spec/)** для компонентного тестирования экранов и **[api](./api/)** для тестирования бэкенда.
* **💼 Менеджеры проектов / Продуктологи:**
  Ваши основные инструменты — **[Шаг 0](./0-customer-brief/)**, **[Шаг 1](./1-elicitation/)** и **[Шаг 2](./2-requirements/)**. Они позволяют контролировать скоуп и соответствие реализации изначальному брифу.
