# Функциональные требования (Functional Requirements)

## 1. Авторизация и Профиль
| ID | Формулировка | Приоритет | Источник |
| :--- | :--- | :--- | :--- |
| FR-10 | Система должна позволять пользователю авторизоваться по номеру телефона (с использованием SMS/OTP). | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 2 |
| FR-15 | При первой авторизации (регистрации) система должна запрашивать у пользователя ввод его Имени. | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 3 |
| FR-20 | При авторизации/регистрации система должна требовать явное согласие на обработку персональных данных. | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 3 |
| FR-25 | Система должна позволять заполнять информацию об аллергиях в профиле пользователя путем выбора из заранее заданного списка (тегов). | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 1 |
| FR-30 | В личном профиле клиента система должна отображать только активные (предстоящие) записи. Архив посещений не отображается (для MVP). | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 2 |

## 2. Расписание
| ID | Формулировка | Приоритет | Источник |
| :--- | :--- | :--- | :--- |
| FR-40 | Система должна отображать список доступных слотов (ClassProgram) на ближайшие 7 дней по умолчанию. Если слотов нет — показывать Empty State. | Must have | `[domain_model.md](../1-elicitation/domain_model.md)`, Раздел 3 (Просмотр расписания) |
| FR-45 | Система должна предоставлять фильтр дат для поиска классов за пределами базовых 7 дней. | Should have | `[domain_model.md](../1-elicitation/domain_model.md)`, Раздел 3 (Просмотр расписания) |
| FR-50 | Система должна отображать детальную информацию о выбранном слоте расписания (программа, описание, уровень сложности, шеф, адрес, время, базовая стоимость, количество свободных мест). | Must have | `[domain_model.md](../1-elicitation/domain_model.md)`, Раздел 3 (Бронирование) |

## 3. Бронирование и Оплата
| ID | Формулировка | Приоритет | Источник |
| :--- | :--- | :--- | :--- |
| FR-60 | Система должна позволять забронировать место на выбранный слот, создавая запись `Booking` в статусе "Ожидает оплаты". | Must have | `[domain_model.md](../1-elicitation/domain_model.md)`, Раздел 3 (Бронирование) |
| FR-65 | При бронировании система должна позволять выбрать опцию аренды экипировки (с указанием ее стоимости), если `available_equipment_stock` > 0. | Must have | `[domain_model.md](../1-elicitation/domain_model.md)`, Раздел 3 |
| FR-70 | Если `available_equipment_stock` = 0 на выбранный слот, чекбокс аренды должен отображаться, но быть заблокированным с поясняющим текстом о том, что инвентарь уже забронирован. | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 1 |
| FR-75 | Перед подтверждением брони система должна рассчитать и отобразить итоговую сумму к оплате (стоимость слота + стоимость аренды экипировки). | Must have | `[domain_model.md](../1-elicitation/domain_model.md)`, Раздел 3 |
| FR-80 | Сразу после успешного создания брони система должна отображать экран с итоговой суммой, инструкциями и реквизитами для перевода оплаты на карту, а также таймером (1 час) до сгорания брони. | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 3 |
| FR-85 | Система должна позволять клиенту отменить свое активное бронирование. | Must have | `[domain_model.md](../1-elicitation/domain_model.md)`, Раздел 3 (Отмена) |
| FR-90 | При попытке отмены бронирования в день проведения класса, система должна выводить предупреждающее модальное окно о "сгорании" занятия, но с возможностью подтвердить отмену. | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 1 |

## 4. Уведомления и Обратная связь
| ID | Формулировка | Приоритет | Источник |
| :--- | :--- | :--- | :--- |
| FR-100 | После завершения класса система должна предложить клиенту оценить шефа от 1 до 5 звезд (без возможности написать текстовый отзыв). | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 2 |
| FR-105 | Карточка/возможность для выставления оценки шефу должна пропадать через 24 часа после завершения класса. | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 2 |
| FR-110 | Система должна отправлять Push-напоминание клиенту за 24 часа до начала забронированного класса. | Must have | `[requirements_summary.md](../1-elicitation/requirements_summary.md)`, Раздел 3 |
| FR-115 | При получении push-уведомления об отмене класса со стороны студии, система должна блокировать возможность повторного бронирования данного отмененного слота. | Must have | `[domain_model.md](../1-elicitation/domain_model.md)`, Раздел 3 (Обработка форс-мажора) |
