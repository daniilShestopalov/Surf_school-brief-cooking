# Сценарии использования (Use Cases)

## [UC-10](./use-cases.md): Авторизация по номеру телефона
* **Актёры:** Клиент
* **Предусловия:** Приложение установлено, клиент не авторизован.
* **Триггер:** Пользователь пытается войти в систему (нажимает кнопку "Войти").

**Контекстные бизнес-решения:**
> Авторизация осуществляется исключительно по номеру телефона с использованием SMS/OTP-кода [[FR-10](./functional-requirements.md)]. Обязательное получение явного согласия на обработку персональных данных (ПД) до этапа авторизации [[FR-20](./functional-requirements.md)].

**Основной поток:**
1. Пользователь вводит свой номер телефона на экране входа.
2. Система предлагает ознакомиться и отметить чекбокс согласия на обработку ПД [[FR-20](./functional-requirements.md)].
3. Пользователь отмечает чекбокс и нажимает "Получить код".
4. Система отправляет запрос к API для генерации и отправки OTP-кода [[NFR-10](./non-functional-requirements.md)].
5. Пользователь вводит полученный из SMS код на следующем экране.
6. Система проверяет код на стороне бэкенда [[NFR-20](./non-functional-requirements.md)].
7. (Только для новых клиентов) Приложение перенаправляет на экран заполнения профиля: пользователь вводит свое Имя и нажимает "Продолжить" [[FR-15](./functional-requirements.md)].
8. При успешной проверке клиент авторизуется в приложении, происходит перенаправление на главный экран.

**Альтернативные потоки:**
* **A1:** Пользователь не отметил согласие на ПД. Кнопка "Получить код" заблокирована, либо при её нажатии система подсвечивает невыбранный чекбокс [[FR-20](./functional-requirements.md)].
* **A2:** Пользователь не получил код и нажимает "Запросить повторно" по истечении таймаута интерфейса.

**Исключения / ошибки:**
* **E1:** Введен неверный OTP-код. Система просит ввести код заново.
* **E2:** Отсутствие сети (Offline). Система не может отправить запрос и уведомляет пользователя [[NFR-10](./non-functional-requirements.md)].

**Матрица ошибок ключевых операций:**
| Код | HTTP | Обязательные details | UX-реакция | Retry |
| :--- | :--- | :--- | :--- | :--- |
| `INVALID_OTP` | 400 | - | Подсветка поля ввода: "Неверный код" | Да |
| `OTP_EXPIRED` | 400 | - | Снекбар: "Код устарел, запросите новый" | Да |
| `NETWORK_ERR` | - | - | Снекбар: "Проверьте подключение к интернету" | Да |

**Постусловия:**
* **При успехе:** Сессия клиента создана, происходит перенаправление на главный экран расписания.
* **При ошибке:** Пользователь остается на экране ввода/запроса кода.

## [UC-20](./use-cases.md): Просмотр расписания слотов
* **Актёры:** Клиент
* **Предусловия:** Клиент авторизован [[FR-10](./functional-requirements.md)].
* **Триггер:** Клиент открывает приложение (главный экран).

**Контекстные бизнес-решения:**
> По умолчанию отображается расписание только на ближайшие 7 дней [[FR-40](./functional-requirements.md)]. Должен присутствовать фильтр для поиска классов за пределами базового периода [[FR-45](./functional-requirements.md)].

**Основной поток:**
1. Приложение запрашивает у бэкенда список доступных слотов на ближайшие 7 дней [[NFR-10](./non-functional-requirements.md)].
2. Система отображает список слотов с базовой информацией.
3. Пользователь нажимает на интересующий слот для просмотра детальной информации (программа, сложность, описание, биография шефа, адрес, базовая стоимость, свободные места) [[FR-50](./functional-requirements.md)].
4. (Опционально) Пользователь использует календарный фильтр дат для загрузки слотов за другие периоды [[FR-45](./functional-requirements.md)].

**Альтернативные потоки:**
* **A1 (Empty State):** В запрашиваемом периоде (ближайшие 7 дней или выбранный диапазон в фильтре) нет доступных слотов. Система отображает иллюстрацию и текст заглушки (Empty State) [[FR-40](./functional-requirements.md)].

**Исключения / ошибки:**
* **E1:** Сбой сети при загрузке расписания. Система выводит кэшированные данные (если есть) или экран критической ошибки соединения.

**Матрица ошибок ключевых операций:**
| Код | HTTP | Обязательные details | UX-реакция | Retry |
| :--- | :--- | :--- | :--- | :--- |
| `FETCH_ERROR` | 500/503 | - | Плейсхолдер: "Не удалось загрузить расписание" + кнопка "Обновить" | Да |

**Постусловия:**
* **При успехе:** Пользователь видит актуальный список слотов или корректное состояние пустоты.
* **При ошибке:** Пользователь видит сообщение о невозможности загрузки.

## [UC-30](./use-cases.md): Бронирование слота с арендой инвентаря
* **Актёры:** Клиент
* **Предусловия:** Клиент авторизован. Выбран конкретный слот, в котором есть свободные места [[FR-50](./functional-requirements.md)].
* **Триггер:** Пользователь нажимает кнопку "Забронировать" на карточке слота.

**Контекстные бизнес-решения:**
> Разрешение на онлайн-бронирование [[BR-10](./business-requirements.md)]. Бэкенд гарантирует строгую атомарную защиту от двойных бронирований [[NFR-20](./non-functional-requirements.md)]. Аренда инвентаря (доп. услуга) [[BR-30](./business-requirements.md)] возможна только при наличии остатка в прокатном фонде [[FR-65](./functional-requirements.md), [FR-70](./functional-requirements.md)]. Эквайринга в MVP нет, оплата осуществляется прямым переводом [[NFR-40](./non-functional-requirements.md)], при этом за клиентом держится бронь на 1 час (статус "Ожидает оплаты") [[BR-40](./business-requirements.md)].

**Основной поток:**
1. Система отображает экран подтверждения деталей бронирования со стоимостью слота.
2. Система проверяет `available_equipment_stock`. Так как инвентарь доступен (> 0), чекбокс аренды активен, рядом указана стоимость [[FR-65](./functional-requirements.md)].
3. Пользователь отмечает чекбокс аренды инвентаря.
4. Система рассчитывает и показывает итоговую сумму к оплате [[FR-75](./functional-requirements.md)].
5. Пользователь нажимает "Подтвердить запись".
6. Приложение отправляет запрос на бэкенд для создания записи `Booking` [[FR-60](./functional-requirements.md)].
7. Бэкенд атомарно резервирует рабочее место и единицу инвентаря, присваивая статус "Ожидает оплаты" [[NFR-20](./non-functional-requirements.md)].
8. При успехе система переводит пользователя на специальный экран с итоговой суммой, инструкциями по оплате, банковскими реквизитами и таймером на 1 час [[FR-80](./functional-requirements.md)].

**Альтернативные потоки:**
* **A1 (Инвентарь исчерпан):** Если `available_equipment_stock` = 0, чекбокс аренды отображается, но жестко заблокирован с поясняющим текстом о недоступности инвентаря [[FR-70](./functional-requirements.md)]. Пользователь продолжает бронирование без опции аренды.

**Исключения / ошибки:**
* **E1 (Гонка за места):** В момент нажатия кнопки "Подтвердить", последнее место было занято другим клиентом. Бэкенд возвращает ошибку валидации атомарной транзакции [[NFR-20](./non-functional-requirements.md)].
* **E2 (Гонка за инвентарь):** Места есть, но выбранный в UI инвентарь закончился на бэкенде к моменту запроса.

**Матрица ошибок ключевых операций:**
| Код | HTTP | Обязательные details | UX-реакция | Retry |
| :--- | :--- | :--- | :--- | :--- |
| `SLOT_FULL` | 409 | - | Модальное окно: "Извините, места на этот класс только что закончились" | Нет |
| `EQUIPMENT_OUT` | 409 | - | Снекбар: "Инвентарь уже забронирован другим. Продолжить без аренды?" | Да (после снятия чекбокса) |

**Постусловия:**
* **При успехе:** Запись `Booking` создана. Количество свободных мест слота уменьшено. Пользователь видит реквизиты.
* **При ошибке:** Бронь не создана, состояние слота не изменено.

## [UC-40](./use-cases.md): Заполнение медицинского профиля (Аллергии)
* **Актёры:** Клиент
* **Предусловия:** Клиент авторизован.
* **Триггер:** Клиент открывает раздел настроек профиля.

**Контекстные бизнес-решения:**
> Аллергии заполняются клиентом один раз в глобальном профиле путем выбора из заранее заданного словаря тегов [[FR-25](./functional-requirements.md)].

**Основной поток:**
1. Пользователь открывает раздел управления здоровьем/аллергиями в профиле.
2. Система загружает справочник доступных аллергенов (тегов) с бэкенда [[NFR-10](./non-functional-requirements.md)].
3. Пользователь отмечает нужные теги из списка.
4. Пользователь нажимает "Сохранить".
5. Приложение отправляет запрос PATCH/PUT для обновления `allergy_profile` сущности `Client` [[NFR-20](./non-functional-requirements.md)].

**Альтернативные потоки:**
* Отсутствуют (прямолинейный процесс сохранения массива ID).

**Исключения / ошибки:**
* **E1:** Сбой сохранения из-за отсутствия интернет-соединения.

**Матрица ошибок ключевых операций:**
| Код | HTTP | Обязательные details | UX-реакция | Retry |
| :--- | :--- | :--- | :--- | :--- |
| `SAVE_ERROR` | 500/503 | - | Снекбар: "Не удалось сохранить изменения. Попробуйте позже" | Да |

**Постусловия:**
* **При успехе:** Настройки профиля обновлены, теги аллергий сохранены и будут автоматически прикрепляться к будущим `Booking`.

## [UC-50](./use-cases.md): Отмена бронирования (Заблаговременная / Поздняя)
* **Актёры:** Клиент
* **Предусловия:** У клиента есть активное предстоящее бронирование, которое отображается в профиле [[FR-30](./functional-requirements.md)].
* **Триггер:** Пользователь открывает детали активной записи и нажимает "Отменить запись" [[FR-85](./functional-requirements.md)].

**Контекстные бизнес-решения:**
> Строгая политика управления отменами для минимизации потерь бизнеса [[BR-20](./business-requirements.md)]. При попытке поздней отмены (в день проведения) пользователь должен получить строгое предупреждение о "сгорании" средств, но сохранить возможность отменить бронь для освобождения места [[FR-90](./functional-requirements.md)]. Вся расчетная логика сроков — на бэкенде [[NFR-20](./non-functional-requirements.md)].

**Основной поток (Заблаговременная отмена):**
1. Пользователь нажимает "Отменить запись" внутри карточки брони.
2. Приложение запрашивает бэкенд на проверку условий отмены для данного `Booking` [[NFR-20](./non-functional-requirements.md)].
3. Бэкенд возвращает статус "Заблаговременная отмена" (до 00:00 дня класса).
4. Приложение показывает стандартное окно: "Вы уверены, что хотите отменить запись? Штраф не взимается.".
5. Пользователь подтверждает действие.
6. Запрос на отмену отправляется в API. Бронь меняет статус на "Отменена клиентом".

**Альтернативные потоки:**
* **A1 (Поздняя отмена):** На шаге 3 бэкенд определяет отмену как "позднюю". Приложение выводит предупреждающее модальное окно с кастомным текстом: "К сожалению, продукты уже закуплены, оплата сгорит. Вы всё равно хотите отменить запись?" [[FR-90](./functional-requirements.md)]. Пользователь подтверждает отмену. Бронь отменяется.

**Исключения / ошибки:**
* **E1:** Попытка отменить бронь, которая уже перешла в статус "Завершена" или была отменена студией в ту же секунду.

**Матрица ошибок ключевых операций:**
| Код | HTTP | Обязательные details | UX-реакция | Retry |
| :--- | :--- | :--- | :--- | :--- |
| `INVALID_STATE` | 400 | current_status | Снекбар: "Невозможно отменить запись в её текущем статусе" | Нет |

**Постусловия:**
* **При успехе:** Запись отменена, рабочее место в слоте высвобождается. Запись скрывается из списка активных в профиле [[FR-30](./functional-requirements.md)].
* **При ошибке:** Статус записи не меняется.

## [UC-60](./use-cases.md): Выставление оценки шефу
* **Актёры:** Клиент
* **Предусловия:** Клиент посетил класс (статус `Booking` = Завершена). С момента завершения прошло менее 24 часов [[FR-105](./functional-requirements.md)].
* **Триггер:** Пользователь получает Push-напоминание [[NFR-30](./non-functional-requirements.md)] и переходит по нему, либо открывает приложение.

**Контекстные бизнес-решения:**
> Сбор пользовательских оценок для рейтинга шефов [[BR-50](./business-requirements.md)]. Оценка осуществляется только звездами (1-5) без текстового отзыва [[FR-100](./functional-requirements.md)]. Окно для выставления оценки закрывается ровно через 24 часа [[FR-105](./functional-requirements.md)].

**Основной поток:**
1. Приложение отображает специальную карточку (или модальное окно) с предложением оценить завершенный класс и работу шефа [[FR-100](./functional-requirements.md)].
2. Пользователь выбирает количество звезд (от 1 до 5).
3. Приложение отправляет запрос с оценкой на бэкенд.
4. Карточка оценки немедленно скрывается из интерфейса.

**Альтернативные потоки:**
* **A1 (Истечение таймаута интерфейса):** Пользователь игнорирует Push и заходит в приложение спустя 24 часа после класса. Бэкенд не возвращает карточку для оценки. Оценить шефа больше нельзя [[FR-105](./functional-requirements.md)].

**Исключения / ошибки:**
* **E1 (Гонка таймаута):** Пользователь открыл карточку оценки на 23:59:50, но нажал отправить на 24:00:05. Бэкенд отклоняет запрос из-за истечения 24-часового окна [[NFR-20](./non-functional-requirements.md), [FR-105](./functional-requirements.md)].

**Матрица ошибок ключевых операций:**
| Код | HTTP | Обязательные details | UX-реакция | Retry |
| :--- | :--- | :--- | :--- | :--- |
| `RATING_TIMEOUT`| 403 | - | Снекбар: "Время для выставления оценки вышло" + скрытие UI карточки | Нет |

**Постусловия:**
* **При успехе:** Оценка сохранена в бэкенде, значение влияет на общий рейтинг шефа [[BR-50](./business-requirements.md)].
* **При ошибке:** Оценка не сохранена.

## [UC-70](./use-cases.md): Получение уведомления об отмене студией
* **Актёры:** Система (Бэкенд), Клиент
* **Предусловия:** У клиента есть активное бронирование на предстоящий слот. Студия инициирует форс-мажорную отмену.
* **Триггер:** Бэкенд принудительно меняет статус слота и рассылает Push-уведомления [[BR-60](./business-requirements.md), [NFR-30](./non-functional-requirements.md)].

**Контекстные бизнес-решения:**
> Своевременное информирование при форс-мажоре со стороны студии [[BR-60](./business-requirements.md)]. Полный запрет на повторное бронирование слота, который был отменен студией [[FR-115](./functional-requirements.md)]. В качестве канала информирования используются исключительно Push-уведомления [[NFR-30](./non-functional-requirements.md)].

**Основной поток:**
1. На устройство клиента приходит системное Push-уведомление с указанием причины отмены класса [[NFR-30](./non-functional-requirements.md)].
2. Клиент нажимает на уведомление и открывает приложение.
3. Приложение запрашивает актуальный стейт с бэкенда. Статус брони меняется на "Отменена студией".
4. Если клиент переходит в расписание, данный слот отображается как "Отменен" и недоступен для повторного бронирования (кнопка заблокирована) [[FR-115](./functional-requirements.md)].

**Альтернативные потоки:**
* **A1 (Push не доставлен):** Из-за системных настроек ОС или отсутствия сети клиент не получил Push. При следующем запуске приложения данные синхронизируются [[NFR-10](./non-functional-requirements.md)], клиент видит изменение статуса брони в профиле.

**Исключения / ошибки:**
*В данном процессе фоновой синхронизации и получения Push-уведомлений прямых ошибок, требующих реакции пользователя (Retry), нет, кроме общих проблем с сетью при загрузке данных (см. [UC-20](./use-cases.md)).*

**Постусловия:**
* **При успехе:** Клиент проинформирован об отмене, статус `Booking` обновлен, место более недоступно для резерва.

## [UC-80](./use-cases.md): Получение Push-напоминания о классе
* **Актёры:** Система (Бэкенд), Клиент
* **Предусловия:** У клиента есть активное бронирование на предстоящий слот (статус `Booking` = Активна). До начала слота остается 24 часа.
* **Триггер:** Наступление времени (T - 24ч) на сервере.

**Контекстные бизнес-решения:**
> Напоминание клиентам для снижения процента неявок [[BR-60](./business-requirements.md)]. В качестве канала используются исключительно Push-уведомления [[NFR-30](./non-functional-requirements.md)].

**Основной поток:**
1. Бэкенд формирует и отправляет Push-уведомление клиенту с текстом-напоминанием ("Ждем вас завтра в HH:MM на класс <название>") [[FR-110](./functional-requirements.md)].
2. Клиент получает уведомление на свое устройство.
3. При нажатии на уведомление открывается приложение (Экран деталей предстоящей брони).

**Альтернативные потоки:**
* **A1 (Push не доставлен):** Из-за системных настроек ОС или отсутствия сети клиент не получил Push. Приложение полагается на то, что клиент сам проверит раздел "Мои записи".

**Исключения / ошибки:**
* Отсутствуют.

**Постусловия:**
* **При успехе:** Уведомление доставлено.
