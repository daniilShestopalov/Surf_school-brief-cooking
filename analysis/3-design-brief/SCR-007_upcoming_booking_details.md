# SCR-007: Детали активной брони

## 1. Контекст и Цель пользователя
- **Зачем пользователь приходит на этот экран?** Вспомнить подробности предстоящего класса (куда ехать, что взял в аренду, оплачено ли), либо перейти сюда по Push-напоминанию за 24 часа или по Push-уведомлению об отмене со стороны студии (US-150, US-160).
- **С каких экранов сюда можно попасть?** Из списка записей в Профиле (SCR-006) при клике на карточку записи, либо прямой переход из Push-уведомлений (UC-70, UC-80).
- **Куда можно уйти отсюда?** На экран инструкции по оплате (SCR-005, если статус "Ожидает оплаты"), на шторку отмены брони (BS-003) или назад.
- **Связанные сценарии:** UC-70, UC-80.

## 2. Информационная Архитектура (Что должно быть на экране)
- **Первичный фокус (High Priority):** Статус бронирования ("Ожидает оплаты", "Оплачено", "Отменено"), дата, время и название класса. Кнопка "Оплатить" (переход к реквизитам), если не оплачено.
- **Вторичный фокус (Medium Priority):** Точный адрес проведения (с картой), список дополнительных услуг (аренда инвентаря: да/нет).
- **Третичный фокус (Low Priority):** Правила отмены (текст), кнопка/ссылка "Отменить запись" (вызов BS-003).

## 3. Матрица Состояний Интерфейса (UX States)
- **Default State:** Отображение деталей брони, кнопка отмены доступна.
- **Loading State:** Скелетон данных при загрузке актуального статуса с бэкенда (особенно важно при переходе из Push).
- **Empty State:** Не применимо.
- **Error State:** Снекбар "Не удалось загрузить данные" с кнопкой повтора.
- **Validation / Invalid State:** Если класс был отменен студией (UC-70), выводится красный баннер "Класс отменен", кнопка оплаты скрывается, статус "Отменено".

## 4. Ограничения и Бизнес-логика для UI
- Если статус брони "Ожидает оплаты", обязательно должен присутствовать таймер и кнопка "Оплатить", ведущая на SCR-005.
- Переход по Push-напоминанию за 24 часа (FR-110) должен открывать именно этот экран, минуя все промежуточные списки.
