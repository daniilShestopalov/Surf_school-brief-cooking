# BS-003: Отмена бронирования (Модальное окно)

## 1. Контекст и Цель пользователя
- **Зачем пользователь приходит на этот экран?** Пользователь инициировал отмену своей записи, система требует подтверждения действия и информирует о последствиях (US-110, US-120).
- **С каких экранов сюда можно попасть?** Вызов из карточки активной записи в Профиле (SCR-006).
- **Куда можно уйти отсюда?** Закрыть окно (отмена отмены) или подтвердить отмену (возврат в профиль с удалением записи).
- **Связанные сценарии:** UC-50.

## 2. Информационная Архитектура (Что должно быть на экране)
- **Первичный фокус (High Priority):** Основной текст предупреждения. Кнопки "Да, отменить запись" (деструктивная, красная) и "Не отменять" (Primary/Secondary) (FR-85, FR-90).
- **Вторичный фокус (Medium Priority):** Заголовок модального окна.
- **Третичный фокус (Low Priority):** -

## 3. Матрица Состояний Интерфейса (UX States)
- **Default State:** 
  - **Вариант 1 (Заблаговременная отмена):** Стандартный текст: "Вы уверены, что хотите отменить запись? Штраф не взимается." (BR-20).
  - **Вариант 2 (Поздняя отмена):** Строгий текст-предупреждение: "К сожалению, продукты уже закуплены, оплата сгорит. Вы всё равно хотите отменить запись?" (FR-90, BR-20).
- **Loading State:** Индикатор загрузки на деструктивной кнопке при запросе к API.
- **Empty State:** Не применимо.
- **Error State:** Снекбар "Невозможно отменить запись в её текущем статусе" (INVALID_STATE) (UC-50).
- **Validation / Invalid State:** Не применимо.

## 4. Ограничения и Бизнес-логика для UI
- Текст предупреждения должен динамически меняться в зависимости от ответа бэкенда при проверке возможности отмены.
- Окно должно явно привлекать внимание (Alert Dialog или Bottom Sheet), чтобы пользователь не мог отменить бронь случайным тапом.
