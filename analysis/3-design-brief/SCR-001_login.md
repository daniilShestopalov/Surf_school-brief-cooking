# SCR-001: Экран входа

## 1. Контекст и Цель пользователя
- **Зачем пользователь приходит на этот экран?** Авторизация в приложении по номеру телефона для доступа к расписанию и бронированию (US-10).
- **С каких экранов сюда можно попасть?** Точка входа при запуске неавторизованного приложения.
- **Куда можно уйти отсюда?** На экран заполнения имени при регистрации (SCR-002) или на главный экран расписания (SCR-003).
- **Связанные сценарии:** UC-10.

## 2. Информационная Архитектура (Что должно быть на экране)
- **Первичный фокус (High Priority):** Поле ввода номера телефона, кнопка "Получить код" (FR-10).
- **Вторичный фокус (Medium Priority):** Поле для ввода OTP-кода (появляется после запроса), кнопка повторного запроса кода (UC-10).
- **Третичный фокус (Low Priority):** Чекбокс явного согласия на обработку персональных данных со ссылкой на документ (FR-20, US-30).

## 3. Матрица Состояний Интерфейса (UX States)
- **Default State:** Пустое поле ввода номера телефона, чекбокс согласия (не выбран по умолчанию), заблокированная кнопка "Получить код".
- **Loading State:** Спиннер на кнопке отправки при запросе OTP и проверке кода.
- **Empty State:** Не применимо.
- **Error State:** Снекбар "Проверьте подключение к интернету" при NETWORK_ERR. Снекбар "Код устарел" при OTP_EXPIRED (UC-10).
- **Validation / Invalid State:** Подсветка поля ввода телефона при некорректном формате; подсветка чекбокса, если он не выбран при попытке отправить; подсветка поля OTP с текстом "Неверный код" при INVALID_OTP (UC-10).

## 4. Ограничения и Бизнес-логика для UI
- Авторизация осуществляется исключительно по номеру телефона (СМС/OTP).
- Кнопка "Получить код" должна быть неактивна или вызывать ошибку валидации, если не отмечен чекбокс ПД (FR-20).
- Кнопка "Запросить повторно" (для OTP) становится активной только по истечении таймаута (например, 60 секунд).
